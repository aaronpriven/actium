8. Service change bags for major changes

While there is a program, “actium.pl bags2”, for making service change bags for major changes, it’s not as regularized as some of the others, and (unlike most of these programs) I’ve usually ended up making changes to the code each time I’ve had to use it. Part of that is because service changes of large magnitude don’t happen that often.

Process

In general, the way it works is like this. The process starts with the output from the “actium.pl comparestops” program. The “Added,” “Removed,” and “Unchanged” columns are used to determine the bag’s basic message of the line changes: which lines stop there, which lines will stop there, which lines will cease to stop there. There may be text associated with each stop giving directions: something like “For downtown Oakland, board on International Blvd.”

It gets complicated because we have to provide Spanish and Chinese translations for everything. So instead of specifying the text itself, all the text is given a brief identifying tag, and the translation is pulled from the “i18n” FileMaker table.

The i18n table is populated by creating an InDesign table with four columns: the first is the tag, the second is the text in English, and the third and fourth are populated by AC Transit’s translation company with the translations in Spanish and Chinese. That InDesign table is saved as a tab-delimited text file, run through the program actium.pl indd_encode, and then imported into FileMaker.

Then, the actium.pl bags2 program creates the file that is the content of the bags, and the “Make Bags 2018.scpt” takes the output from actium.pl bags2 and creates bag decals, using the file PubInfSys – Documents/Flags/Bags/Generated_bag_template_2019.indd”

There are a lot of specifics, but the truth is that I have to program them every time. The program isn’t usable, now, by someone who isn’t comfortable doing that.

Future

Tony manually creates tons of service change bags and I presume that will continue. There probably won’t be any need for large numbers of service change bags until the new network is built.

I don’t think I really thought about this program reinventing the “mail merge” / “data merge” wheel when I first created it. Probably the whole thing should be thrown out and reimplemented using “data merge” in InDesign, which shouldn’t require the same level of programming as this does.
